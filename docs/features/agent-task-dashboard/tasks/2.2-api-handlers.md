# 2.2: API Handlers — REST Endpoints

## Description

Implement all 7 REST API endpoints that serve JSON data to both the frontend and agent clients.

## Reference Files

- `design/tech-design.md` — Interface 4: API Handlers
- `design/api-handbook.md` — All endpoint specifications with request/response formats
- `design/tech-design.md` — DependencyGraph model (for `/dependencies` endpoint)

## Acceptance Criteria

- [ ] `internal/handler/api.go` implements `RegisterAPI(r *gin.Engine, scanner *Scanner)`
- [ ] `GET /api/projects` returns array of project summaries with computed stats
- [ ] `GET /api/projects/:id` returns single project with feature list
- [ ] `GET /api/projects/:id/features` returns feature array with task counts
- [ ] `GET /api/projects/:id/features/:slug` returns feature with tasks and phases
- [ ] `GET /api/projects/:id/features/:slug/tasks` returns task array with all fields
- [ ] `GET /api/projects/:id/features/:slug/tasks/:taskId` returns full task with `acceptanceCriteria` and `executionRecord`
- [ ] `GET /api/projects/:id/features/:slug/dependencies` returns `DependencyGraph` with nodes and expanded edges (wildcards resolved)
- [ ] All responses include `meta.lastUpdated` timestamp
- [ ] 404 returned for non-existent project/feature/task IDs
- [ ] `completionPct` is 0 when `totalTasks` is 0 (no division by zero)
- [ ] Integration tests using `httptest` + Gin test mode: all endpoints happy path + 404s
- [ ] All tests pass

## User Stories

- **Story 6**: Agent queries task state via API → all 7 endpoints provide queryable data

## Implementation Notes

- Each handler calls Scanner methods, formats response per api-handbook.md
- The `/dependencies` endpoint must resolve wildcard deps ("1.x") to individual edges
- Cross-feature edges should have `crossFeature: true` when source and target features differ
- The `/tasks/:taskId` endpoint calls `ParseTaskFile` and `ParseRecordFile` from 2.1
