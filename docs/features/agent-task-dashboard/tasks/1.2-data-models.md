# 1.2: Define Data Models

## Description

Implement all Go data model structs from the tech design. These models are the foundation used by every other component.

## Reference Files

- `design/tech-design.md` — Data Models section (Models 1-8)
- `design/api-handbook.md` — Response field types

## Acceptance Criteria

- [ ] `internal/model/project.go` defines: `ProjectData`, `FeatureData` (with `HasBlockedTasks`, `CompletionPct`), `PhaseInfo`
- [ ] `internal/model/task.go` defines: `Task` with all fields including `Phase` (derived from ID)
- [ ] `internal/model/graph.go` defines: `DependencyGraph`, `GraphNode`, `GraphEdge` (with `CrossFeature`)
- [ ] `internal/model/activity.go` defines: `ActivityEvent`
- [ ] `internal/model/record.go` defines: `RecordContent` (Summary, Files, Decisions, TestResults, Raw), `TaskFileContent`
- [ ] `internal/model/errors.go` defines error types: `ErrProjectNotFound`, `ErrFeatureNotFound`, `ErrTaskNotFound`, `ErrConfigInvalid`, `ErrFSRead`, `ErrParseIndex`, `ErrInvalidSlug`, `ErrInvalidTaskID`
- [ ] All structs have JSON tags matching api-handbook.md response format
- [ ] `Phase` field has a helper function `DerivePhase(id string) int` that extracts the leading number
- [ ] All code compiles without errors

## User Stories

No direct user story mapping — data layer task.

## Implementation Notes

- Error types should implement the `error` interface with `Error()` method
- JSON tags must match api-handbook.md exactly (e.g., `json:"completionPct"`, `json:"crossFeature"`)
- `DerivePhase` should handle special cases: "T-test-1" → phase 0 (testing), non-numeric IDs → phase 0
