# 1.5: Error Handling and Input Validation

## Description

Implement error propagation middleware and input validation for URL parameters. Ensures consistent error responses across all API endpoints.

## Reference Files

- `design/tech-design.md` — Error Handling section (Error Types, Propagation Strategy)
- `design/api-handbook.md` — Error Codes table

## Acceptance Criteria

- [ ] `internal/handler/middleware.go` implements validation middleware for `:id`, `:slug`, `:taskId` path params
- [ ] Slug validation: `^[a-z0-9][a-z0-9-]*[a-z0-9]$` (alphanumeric + hyphens, 2+ chars)
- [ ] Task ID validation: `^[a-zA-Z0-9._-]+$` (no path traversal sequences like `..`)
- [ ] Invalid input returns HTTP 400 with `{"error": "invalid_slug", "message": "..."}` or `{"error": "invalid_task_id", ...}`
- [ ] Gin error middleware catches scanner errors and maps to appropriate HTTP status codes (404, 500)
- [ ] Error response format consistent: `{"error": "<code>", "message": "<description>"}`
- [ ] Unit tests: valid slug passes, `../` rejected, spaces rejected, valid task IDs pass, `..` rejected
- [ ] All tests pass

## User Stories

- **Story 6**: Agent receives proper 404 for non-existent IDs → error propagation maps correctly

## Implementation Notes

- Validation runs in Gin middleware before handler is called
- Use regex compilation at package init time (not per-request)
- Internal error codes (ERR_INVALID_SLUG) map to external codes (invalid_slug) in the middleware
